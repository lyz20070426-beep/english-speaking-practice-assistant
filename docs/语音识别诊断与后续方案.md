# 语音识别诊断与后续方案

## 当前问题判断

当前免费版朗读评分使用浏览器的 `SpeechRecognition` 能力。该能力在不同浏览器、不同系统、不同网络环境下表现不稳定。

根据 MDN 文档，`SpeechRecognition` 属于有限可用能力，并不是所有主流浏览器都稳定支持。MDN 也说明，在 Chrome 等浏览器中，语音识别可能依赖服务端识别引擎，音频会发送到网页服务进行处理，因此离线或网络受限时可能无法工作。

参考：

- https://developer.mozilla.org/en-US/docs/Web/API/SpeechRecognition
- https://developer.mozilla.org/en-US/docs/Web/API/Web_Speech_API

## 如何侧面验证 SpeechRecognition 是否正常

项目已新增诊断页：

```text
app/speech-diagnostic.html
```

GitHub Pages 发布后访问：

```text
https://lyz20070426-beep.github.io/english-speaking-practice-assistant/app/speech-diagnostic.html
```

诊断页会检查：

- 浏览器是否存在 `SpeechRecognition` 或 `webkitSpeechRecognition`
- 麦克风权限是否正常
- 是否触发 `audiostart`
- 是否触发 `speechstart`
- 是否触发 `result`
- 是否触发 `error`
- 是否触发 `end`

如果能看到 `event:result`，说明浏览器语音识别正常返回了结果。

如果只有 `audiostart`、`speechstart`，但没有 `result`，说明麦克风可以用，但浏览器语音识别服务没有返回识别文本。

如果出现 `event:error`，需要记录错误值，例如：

- `not-allowed`：麦克风权限被拒绝
- `no-speech`：没有检测到语音
- `network`：语音识别服务网络异常
- `service-not-allowed`：浏览器或系统不允许使用语音识别服务

## 更好的解决方案

### 方案 A：继续使用浏览器免费能力

优点：

- 0 元成本
- 不需要服务器
- 适合演示和早期验证

缺点：

- 依赖浏览器兼容性
- Edge、Safari、手机端可能不稳定
- 无法做到专业发音评分

适合当前原型阶段。

### 方案 B：浏览器录音 + 手动/半自动评分

优点：

- 录音功能比语音识别更稳定
- 老师可以听录音
- 系统可以保留学生练习证据

缺点：

- 自动评分能力弱
- 老师工作量较大

适合短期试用和教学反馈。

### 方案 C：接入专业语音识别或发音评测接口

优点：

- 稳定性更高
- 可支持更真实的发音评分
- 可返回准确度、流利度、完整度等维度

缺点：

- 通常需要费用
- 需要后端服务器或云函数保护密钥
- 需要处理学生音频数据和隐私

可选方向包括讯飞、腾讯云、阿里云、Azure Speech 等。

### 方案 D：本地模型或离线识别

优点：

- 数据不出本地
- 可减少云端费用

缺点：

- 部署复杂
- 学校电脑性能和环境不一定适合
- 开发周期明显增加

不建议作为 1.0 版优先方案。

## 建议路线

短期：

1. 使用 GitHub Pages HTTPS 地址测试。
2. 用诊断页分别测试 Chrome、Edge、手机浏览器。
3. 保留手动评分兜底，保证老师演示不断。

中期：

1. 如果老师只需要练习记录，可以继续免费方案。
2. 如果老师明确要求稳定自动评分，应评估专业发音评测接口。

长期：

1. 增加后端。
2. 上传学生录音。
3. 调用专业发音评测服务。
4. 保存完整评分报告。
