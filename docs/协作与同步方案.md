# 英语口语练习助手协作与同步方案

## 目标

让爸爸在本地电脑上的开发进展、项目文件、需求讨论和女儿在武汉学校电脑上的开发环境保持同步。Codex 本身主要读取当前电脑上的本地文件，因此同步的核心不是共享同一个 Codex 对话，而是共享同一套项目代码和项目文档。

## 推荐协作架构

### 1. GitHub 同步代码和项目文档

建议创建一个 GitHub 私有仓库，例如：

```text
english-speaking-assistant
```

仓库中保存：

- 前端代码
- 后端代码
- 数据库脚本
- 配置说明
- 项目文档
- 需求和开发计划

爸爸在本地提交并推送代码，女儿在学校电脑上拉取最新代码。女儿完成修改后，也通过 GitHub 推送回来。

### 2. 共享文档同步讨论和学校反馈

建议使用飞书文档、腾讯文档、Notion、语雀或 Google Docs 记录更偏业务和沟通的内容：

- 学校老师提出的需求
- 女儿试用后的反馈
- 产品功能想法
- 阶段性会议记录
- 待办事项

这些内容可以定期整理进本项目的 `docs` 目录，方便 Codex 在本地读取上下文。

### 3. Codex 辅助各自本地开发

爸爸和女儿可以分别在自己的电脑上使用 Codex。Codex 可以帮助：

- 阅读项目代码
- 修改功能
- 整理需求文档
- 生成开发计划
- 排查报错
- 编写测试

但 Codex 不会天然同步两台电脑的本地文件，所以每次重要修改都要通过 GitHub 进行同步。

## 第一次建立同步

### 爸爸这边

1. 在 GitHub 创建私有仓库。
2. 在本地项目目录初始化 Git。
3. 提交当前文件。
4. 推送到 GitHub。
5. 邀请女儿加入仓库。

常用命令示例：

```bash
git init
git add .
git commit -m "初始化英语口语练习助手项目"
git branch -M main
git remote add origin 仓库地址
git push -u origin main
```

### 女儿这边

1. 登录 GitHub。
2. 接受仓库邀请。
3. 在学校电脑上克隆仓库。
4. 使用 Codex 打开克隆下来的项目目录。

命令示例：

```bash
git clone 仓库地址
cd english-speaking-assistant
```

## 日常同步流程

### 爸爸修改后

```bash
git status
git add .
git commit -m "更新需求文档"
git push
```

### 女儿同步最新内容

```bash
git pull
```

### 女儿修改后

```bash
git status
git add .
git commit -m "完善口语练习页面"
git push
```

### 爸爸同步女儿的修改

```bash
git pull
```

## 分工建议

### 爸爸侧重点

- 整理总体需求
- 维护项目计划
- 协调功能优先级
- 审查代码和文档
- 推进系统整体架构

### 女儿侧重点

- 提供真实学生使用场景
- 收集学校老师和同学反馈
- 测试口语练习体验
- 协助整理练习素材
- 参与前端交互和页面优化

## 需要沉淀到项目里的内容

每次重要讨论后，建议把结论写入项目文件，而不是只留在聊天里。

建议维护这些文件：

- `README.md`：项目介绍、运行方式、当前状态
- `docs/需求说明.md`：系统要解决什么问题、面向谁、有哪些核心功能
- `docs/开发计划.md`：阶段目标、任务拆分、优先级
- `docs/会议记录.md`：每次沟通后的结论
- `docs/问题清单.md`：待确认问题、技术风险、需求疑问

## 推荐规则

1. 每次开始工作前先 `git pull`。
2. 每次完成一小段可说明的工作后就 `git commit`。
3. 每天结束前至少 `git push` 一次。
4. 重要结论不要只放在微信或 Codex 对话里，要写进 `docs`。
5. 两个人尽量不要同时改同一个文件的同一段内容。
6. 遇到 Git 冲突时，先沟通确认保留哪一版，再解决冲突。

## 最小可行协作方式

如果暂时还不熟悉 GitHub，可以先采用过渡方案：

- 代码文件放在 GitHub。
- 讨论和反馈放在腾讯文档或飞书文档。
- 每周把文档里的关键内容整理到本项目 `docs` 目录。

等项目进入正式开发后，再把 GitHub 作为主要同步方式。
